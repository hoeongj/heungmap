# 04. 데이터 & API — "데이터 조립"이 뭔지 + 데이터 스파이크

## 한 줄 개념 (초심자용 핵심)
> **데이터 조립 = 여러 API에서 받은 흩어진 조각을, 공통 키(지역+날짜)로 엮어 표 하나(csv)로 붙이는 것.**
> 그 결과 csv는 ML 대회에서 주최측이 주는 `train.csv`와 똑같은 물건. 이번엔 **우리가 그 표를 직접 만든다.**
> 엑셀 VLOOKUP으로 여러 시트를 한 시트에 합치는 것의 파이썬(pandas) 버전.

## 우리가 만들 최종 학습표 (목적지)
1행 = 축제 1개.

| 축제명 | 지역(시군구) | 시작월 | 기간(일) | 카테고리 | 작년방문자 | 주말수 | 평균기온 | ... | **label: 방문자수** |
|---|---|---|---|---|---|---|---|---|---|

- **features**(입력): 지역·시기·기간·카테고리·과거방문자·날씨·주말포함 등
- **label**(정답): 축제 기간 방문자 수(또는 평상시 대비 증가분 = 흥행도)

## 재료 (어느 API가 표의 어느 칸을 채우나)
| 소스 | 채우는 것 | 비고 |
|---|---|---|
| **TourAPI `searchFestival`** (국문 관광정보 서비스) | 축제 메타: 축제명·시기·기간·지역코드·카테고리·contentid | 필수 활용 대상(심사) |
| **공공데이터포털 "지역별 방문자수"** (한국관광 데이터랩 빅데이터) | **label**: 지역·기간 방문자 시계열 | 통신/카드 기반 |
| **기상청 API** | feature: 축제 기간 날씨 | Tier1에서 추가 |
| (선택) 공연·출연진 인기 지표(외부) | feature: 가수 인기 | 여력 될 때 |

- 발급: 모두 https://data.go.kr 활용신청 → serviceKey. `.env`에 저장(`.env.example` 참조).
- API 참고: https://api.visitkorea.or.kr (TourAPI 가이드), https://datalab.visitkorea.or.kr (데이터랩)

## 조립 5단계
1. **수집(Collect)**: 각 API를 파이썬 `requests`로 호출 → 원본 JSON 저장(`data/raw/`). 연도×지역 루프로 축제 여러 개 모음.
2. **파싱(Parse)**: JSON → pandas DataFrame(표)로 변환.
3. **결합(Join) ← 핵심**: 축제표 + 방문자표를 **공통 키(지역코드+기간)**로 `merge`. 이게 "조립".
4. **파생(Feature engineering)**: 기간=끝-시작, 주말수, 작년 동일축제 방문자 등 새 칼럼 생성.
5. **라벨/정제**: label 칼럼 확정 + 결측/이상치 처리 → `data/processed/train.csv` 저장.

결합 개념(의사코드):
```python
merged = festivals.merge(visitors, on=["area_code", "yyyymm"], how="left")
merged = merged.merge(weather,   on=["area_code", "yyyymm"], how="left")
```
→ 이렇게 나온 `train.csv`를 LightGBM에 먹이면 그게 Tier 1.

## 🚦 데이터 스파이크 (본개발 전 go/no-go 게이트)
**유일한 진짜 리스크 = "축제에 방문자 라벨이 실제로 붙느냐". 며칠 안에 확인하고 결정.**

체크리스트:
- [ ] `TOURAPI_SERVICE_KEY`, `VISITOR_API_SERVICE_KEY` 발급 완료
- [ ] `searchFestival`로 과거 축제 목록 **50~100개** 수집 성공(축제명·지역·기간 확보)
- [ ] 지역별 방문자수 API에서 해당 **지역·기간 방문자 값**을 받아옴
- [ ] 두 표를 지역+기간으로 **join → 라벨이 실제로 매칭되는 행이 충분(수십 개+)**한지 확인
- [ ] (판단) 라벨 품질이 모델 학습에 쓸 만한가?

결과:
- ✅ **통과** → 지정과제 9번 확정, Tier 0 착수.
- ❌ **실패**(라벨 매칭 부족) → **지정과제 2번(관광 혼잡 예측)으로 피벗.** 방문자 시계열이 연속적이라 데이터가 더 안전하고, 같은 예측 근육·포폴 가치를 유지. (`07_DECISION_LOG.md`)

> 첫 세션 팁: 이 스파이크가 팀의 **첫 "데이터 조립" 실습**이기도 함. Claude가 수집→파싱→join 스크립트를 한 단계씩 같이 작성.
