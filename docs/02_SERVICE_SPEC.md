# 02. 서비스 기획서 — 흥할지도 (HeungMap)

## 핵심 컨셉
축제 흥행을 **주관적 경험이 아니라 데이터로 예측**한다. 그 **하나의 예측 엔진**이 두 사용자를 동시에 먹인다.

```
                  ┌─────────────────────────┐
   기획자 입력 ──▶ │  흥행 예측 엔진 (HeungMap)  │ ──▶ 방문객 정보
 (지역·시기·규모·   │  방문자 수 / 흥행 점수 / 근거 │     (혼잡·티켓 경쟁도·
  테마·출연 등)     └─────────────────────────┘      주변 주차/숙소/코스)
```
기획자가 만든 축제가 → 방문객 화면의 정보로 흐르는 **양방향 순환 구조**.

## 타깃
- **기획자(B2B)**: 예산 집행 전 기획안의 객관적 수요 검증이 필요한 지자체·축제 위탁사 실무자.
- **방문객(B2C)**: 축제 방문을 계획하는 일반 관광객.

## 기능 명세

### A. 기획자 모드
1. **축제 조건 입력** — 지역, 개최 시기, 기간, 예상 규모, 테마/카테고리, (선택)출연진·프로그램.
2. **흥행 예측** — 예상 방문자 수 + 흥행 점수(0~100) 도출. *Tier0=규칙기반, Tier1=ML 모델.*
3. **예측 근거 제시** — 어떤 요인이 점수를 올리고/내렸는지. *Tier2=SHAP.*
4. **개최지·편의시설 추천** — 후보지 주변 주차장·숙박·접근성(TourAPI/지도) 기반 최적 입지 제안.
5. **기획 보완 피드백 리포트** — 약점 요인 + 개선안 자동 생성. *Tier2=LLM.*

### B. 방문객 모드
1. **축제 탐색** — 기획자가 등록/공개한 축제 + TourAPI 실제 축제(`searchFestival`) 목록.
2. **예상 혼잡도 / 티켓팅 경쟁도 지표** — 예측 방문자 기반. (※ '리셀 값'은 제외 — `07_DECISION_LOG.md` 참조)
3. **주변 정보** — 주차장·숙소·연계 관광지(TourAPI 위치기반).
4. **방문 코스·즐길거리 추천** — 지도 위 동선.

## 차별점 (심사 독창성 30점 겨냥)
1. 예시 서비스들은 "LLM 추천 1회"에 그침 → 우리는 **자체 학습 예측 모델 + 설명가능성(SHAP)**으로 '주관적 경험 → 데이터 기반 객관 검증'을 실제 구현.
2. **기획자·방문객을 하나의 예측 엔진으로 연결**한 양방향 구조(대부분 단방향).
3. 포폴 관점: 데이터 엔지니어링 + ML + 지도/추천 + LLM + 풀스택을 한 프로젝트에 집약.

## 기대효과
- 기획자: 흥행 실패·예산 낭비 리스크 사전 진단, 개최지·편의 구성 최적화.
- 방문객: 혼잡·티켓팅 불확실성 완화, 방문 만족도↑.
- 지역: 데이터 기반 축제 운영으로 관광 활성화.

## MVP 경계 (반드시 완성해야 하는 최소본 = Tier 0)
- 기획자: 조건 입력 → 규칙기반 흥행 점수 + 후보지 주변 편의시설 표시.
- 방문객: 실제 축제 목록(TourAPI) + 예상 혼잡도 + 주변 주차/숙소 지도.
- 이 범위가 "작동하는 완제품"이며 제출 기준선. 나머지는 그 위 업그레이드.
