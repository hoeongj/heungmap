# 03. 계층형(Tier) 개발 로드맵

## 왜 계층형인가
팀이 초심자라 **새로 배우는 게 3개(데이터 조립·ML·풀스택)**를 동시에 쌓아야 함. 리스크를 줄이려고
**"제품이 항상 작동하는 상태"를 유지**하며 위로 올린다. 모델이 실패해도 프로젝트가 안 망함.

## Tier 구조
| Tier | 내용 | 새로 배우는 것 | 끝나면 |
|---|---|---|---|
| **Tier 0** | 데이터 조립 파이프라인 + **규칙기반 흥행점수(ML 아님, 산수)** + 웹 뼈대(입력→점수→결과) + TourAPI 연동 + 지도(주차/숙소) | API 수집·pandas·웹 기초 | **제출 가능한 완제품** |
| **Tier 1** | 규칙점수를 **LightGBM 회귀 모델**로 교체 (팀의 첫 ML) | 첫 ML 학습/평가 | "데이터 기반 예측" 완성 |
| **Tier 2** | **SHAP** 예측 근거 + **LLM** 보완 리포트 + UI 다듬기 | 설명가능성·LLM 연동 | 대상 노림수 완성본 |

> 핵심 원칙: **Tier 0를 완전히 끝내기 전에는 Tier 1로 넘어가지 않는다.**

## 8주 로드맵 (개발기간 7월 말 ~ 9월)
- **주 0 (며칠, 게이트):** 참가신청 제출 + API 키 발급 + **데이터 스파이크**(→ `04_DATA_AND_APIS.md`). go/no-go.
- **주 1~3: Tier 0**
  - 데이터 수집 스크립트(`searchFestival` + 방문자 API) → 표(csv)로 저장
  - 규칙기반 흥행점수 함수(과거 방문자·주말수·계절·접근성 가중치)
  - 웹 뼈대: 기획자 입력 폼 → 점수 결과, 방문객 축제 목록 → 상세
  - 지도에 주차/숙소 표시(Kakao Map + TourAPI 위치기반)
  - ✅ 여기서 "완성된 서비스" 확보
- **주 4~5: Tier 1**
  - 학습표 정리(feature/label) → LightGBM 학습·검증(교차검증, 지표)
  - 규칙점수 → 모델 예측으로 교체, 성능 비교
- **주 6~7: Tier 2**
  - SHAP로 예측 근거 시각화
  - LLM(Claude)로 "기획 보완 리포트" 생성
  - 방문객 코스 추천·UI 완성도
- **주 8: 마감**
  - 버그·안정화, 데모 시나리오, 1차 심사 기능설명서, PT 자료

## 역할 분담(초안, 2인)
- 데이터/모델 라인(수집→조립→예측): 주로 A
- 웹/화면/지도 라인(프론트+백 연동): 주로 B
- 겹치는 부분은 페어로. 초심자라 **한 명이 막히면 같이** 진행 권장.
> 실제 이름/역할은 다음 세션에서 채우기.

## 스코프 컷 규칙 (시간 부족 시 버리는 순서)
1순위 사수: Tier 0 완제품 + TourAPI 활용. → 그다음 Tier1 모델 → SHAP → LLM 리포트 순으로만 포기.
