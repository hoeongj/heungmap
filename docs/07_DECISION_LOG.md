# 07. 의사결정 기록 (왜 이렇게 정했나)

> 다음 세션이 "왜?"를 다시 묻지 않도록 근거를 남김. (2026-07-11 논의 기준)

## D1. 지정과제 9번 선택 (축제 흥행 예측)
- 후보: 6번(반려동물, 안전), 2번(혼잡 예측, ML), 9번(축제 예측, ML).
- 팀이 **포트폴리오 임팩트 + 차별화**를 우선. 9번은 "예측 모델"이라 대부분 팀이 못 하는 영역 → **독창성·대상 상한선 최고**.
- 예시 10개가 전부 "입력→LLM 추천 1회" 수준이라, **자체 예측 엔진**을 얹는 것이 확실한 차별점.
- 리스크(학습 데이터)는 **데이터 스파이크 게이트 + 계층형 개발**로 관리.

## D2. 계층형(Tier) 개발 채택
- 팀이 초심자(ML·데이터파이프라인·풀스택 모두 처음). 세 가지를 동시에 쌓는 부담.
- 해결: Tier0(규칙기반, 제품 완성 보장) → Tier1(첫 ML) → Tier2(SHAP+LLM). 모델 실패해도 제품이 살아있음.
- "배우면서 도전" 방향을 팀이 명시적으로 선택.

## D3. 양방향 플랫폼 컨셉 (기획자 + 방문객)
- 하나의 예측 엔진으로 B2B(기획자 의사결정 지원) + B2C(방문객 정보) 동시 제공 → 실용성·발전성 점수 + 서사 강화.

## D4. 네이밍
- 서비스명 **흥할지도 (HeungMap)**: "흥할지(도) 몰라" + "지도(map)" 이중 의미. 예측+지도 서비스와 정확히 부합.
- 팀명 **경사하강단**: Gradient Descent 말장난 → ML 팀 정체성.

## D5. "예상 리셀(암표) 값" 기능 제외
- ① 공개 데이터 사실상 부재(구현 곤란) ② 관 주최 공모전에서 암표 조장 오해 리스크.
- 대체: "티켓팅 경쟁도/수요 지표"로 같은 니즈를 안전하게 전달.

## D6. 서비스 유형 = 웹 단일
- 예측 대시보드 + 두 포털 구조에 웹이 최적. 웹+앱 동시는 "동일 내용 2종" 의무라 초심자 팀엔 부담.

## D7. 피벗 플랜 (데이터 스파이크 실패 시)
- 9번 라벨 매칭이 부족하면 **지정과제 2번(관광 혼잡 예측)**으로 전환. 방문자 시계열이 연속적이라 데이터가 더 안전하고, 같은 예측 근육·포폴 가치 유지.
